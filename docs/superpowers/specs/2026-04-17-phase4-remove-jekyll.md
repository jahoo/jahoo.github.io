# Phase 4: Remove Jekyll

**Date:** 2026-04-17
**Status:** Draft
**Depends on:** Phase 3 (Pandoc-driven homepage + publications, complete)

## Goal

Remove Jekyll from the active pipeline. After this phase, the site builds and deploys entirely through the Pandoc + Make toolchain; Jekyll's source lives only in a preserved archive branch for posterity.

## Approach at a glance

Branch strategy:

- `source` (Jekyll-era) → renamed to `source-jekyll-deprecated`. Archive only; no further pushes.
- `pandoc-migration` → renamed to `source`. Becomes the new source-of-truth branch that maintenance pushes to.
- `master` → renamed to `static`. Still the branch GitHub Pages serves from; new name reflects its role.

Deploy:

- Today: push to `source` triggers a GitHub Actions workflow that runs `bundle exec jekyll build` and pushes to `master`.
- After: a new `make deploy` target, run locally, builds the site and pushes `_site/` to `static` from within a git worktree. GitHub Actions is removed.

Local-only deploy solves the `source.bib` problem — the symlink that build-pubs.js depends on lives at `~/all-biblatex.bib` on the user's machine and doesn't exist on a CI runner. Since the maintainer is the only one who pushes, there's no value in a CI rebuild; an explicit `make deploy` is clearer than a git hook.

## Branch operations

Execute in order. Each step is non-destructive (old branches remain reachable as named refs).

1. **Rename `source` → `source-jekyll-deprecated`.**
   - GitHub UI: Settings → Branches → rename `source` to `source-jekyll-deprecated`. Preserves refs and adds an automatic redirect from the old name.
   - Locally: delete the stale local `source` branch if present; fetch.
2. **Rename `master` → `static`.**
   - GitHub UI: rename. Pages' "source branch" setting auto-updates.
   - Locally: `git branch -m master static` on whichever copy you have.
3. **Rename `pandoc-migration` → `source`.**
   - Locally: `git branch -m pandoc-migration source`.
   - Push: `git push origin source` (first-time push sets the upstream).
4. Verify GitHub Pages is serving from `static` (Settings → Pages). It should be — the rename in step 2 should have updated the setting — but worth a visual confirmation.

After these four steps, the remote state is:

- `source` — Pandoc source of truth (what was `pandoc-migration`).
- `static` — deploy output (what was `master`).
- `source-jekyll-deprecated` — archived Jekyll history.

## `make deploy` — the new deployment path

Added to `Makefile`. The sole target that touches the `static` branch.

```make
.PHONY: deploy
deploy:
	@git diff --quiet && git diff --cached --quiet || \
	  { echo "error: uncommitted changes on $$(git branch --show-current); commit or stash first"; exit 1; }
	@[ "$$(git branch --show-current)" = "source" ] || \
	  { echo "error: must be on 'source' branch (currently on $$(git branch --show-current))"; exit 1; }
	@$(MAKE) clean all
	@git push origin source
	@git worktree add _deploy static 2>/dev/null || { \
	  echo "error: _deploy worktree already exists; run 'git worktree remove _deploy' first"; exit 1; \
	}
	@rsync -a --delete --exclude='.git' _site/ _deploy/
	@cd _deploy && git add -A && \
	  ( git diff --cached --quiet && echo "static: no changes to deploy" || \
	    ( git commit -m "Deploy $$(cd .. && git rev-parse --short HEAD)" && \
	      git push origin static ) )
	@git worktree remove _deploy
```

### Behaviour

Preconditions:

- Fails with a clear message if the working tree has uncommitted changes. Deploys reflect `HEAD`, so uncommitted changes would quietly not ship — failing loudly is better.
- Fails if the current branch is not `source`. `make deploy` is only meaningful from that branch.

Steps:

1. `make clean all` — fresh full build into `_site/`. Same as what you get from `make` on a clean checkout, but explicit to ensure no stale files.
2. `git push origin source` — pushes the source commit that backs this deploy. The deploy commit on `static` references it by short-SHA, so history is traceable.
3. `git worktree add _deploy static` — checks `static` out at `_deploy/` without affecting the current working tree. `_deploy/` is git-aware inside (its own `.git` link-file).
4. `rsync -a --delete --exclude='.git' _site/ _deploy/` — mirror the build output into the worktree. `--delete` removes files no longer in the build; `--exclude='.git'` preserves the worktree's git linkage.
5. `cd _deploy && git add -A && ...` — stage everything in the worktree, then commit and push only if there are changes. If the build produced no diff from the last deploy, skip the commit with a "no changes to deploy" message.
6. `git worktree remove _deploy` — tear down the worktree. The `_deploy/` directory is gone after this runs successfully.

Commit message format on `static`: `Deploy <short-sha-of-source>`. Short and identifies the source commit deployed.

### `.gitignore`

Add `_deploy/` so that if a deploy exits mid-way (leaving the worktree behind), its transient presence doesn't pollute `git status` on `source`.
